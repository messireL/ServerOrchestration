# ServerOrchestration v0.1.50 — browser-style subscription HTML parsing

## Что вошло
- сохранён статичный/AJAX-контур без возврата к хрупким frontend-экспериментам;
- усилен backend-парсинг страницы подписки 3x-ui, когда endpoint отдаёт не чистый base64/blob, а HTML-страницу как для браузера;
- добавлен HTML extractor на базе `HTMLParser`:
  - разбор табличных строк `ID подписки / Статус / Загружено / Отправлено / Использование / Общий лимит / Остаток / Был(а) в сети / Срок действия`;
  - fallback-разбор плоского текстового содержимого страницы;
  - fallback-разбор embedded/script-данных, если значения лежат внутри JS/inline payload;
- нормализованы поля профиля подписки:
  - `downloaded_bytes`, `uploaded_bytes`, `used_bytes`, `total_bytes`, `remaining_bytes`;
  - `subscription_id`, `profile_status`, `last_seen_text`, `expires_text`, `expires_unlimited`;
- если страница не отдаёт `used/remaining` напрямую, backend теперь пытается вычислить их из `upload/download/total`.

## Зачем релиз
Нужен именно под сценарий, который ты описал: endpoint подписки открывается как веб-страница, и мониторинг должен обращаться к нему как браузероподобный клиент и вытаскивать данные профиля подписки из HTML, а не тупо упираться в `html-embedded`.

## Что проверить после выкладки
1. Кнопка `Проверить 3x-ui` и `Запустить все проверки` по-прежнему нажимаются без JS-падений.
2. Для сервера с 3x-ui subscription в hover/деталях появляются:
   - usage/used traffic;
   - total limit;
   - remaining;
   - expiry / бессрочность;
   - last seen / статус.
3. В истории прогонов для `xui` нет деградации по сравнению с предыдущим билдом.
4. SSL-проверка продолжает работать отдельно от subscription-парсинга.

## Техническая заметка
Если конкретный subscription endpoint отдаёт данные не в DOM и не в inline-script, а подгружает их отдельным XHR уже после открытия страницы, следующим шагом нужно будет добить точечный разбор именно этого ответа. Но для обычного HTML-профиля 3x-ui этот релиз должен закрыть дыру.
