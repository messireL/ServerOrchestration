# ServerOrchestration v0.1.32

## Что исправлено
- Исправлен критичный backend-crash на старте: из `src.main` убран импорт несуществующей функции `run_xui_check`, из-за которого приложение не поднималось вообще.
- Довосстановлены импорты в probe-контуре: добавлены недостающие `urlparse`, `datetime/timezone`, а также типы `Any` / `Dict` в `main.py`.
- Проверен Python import/runtime smoke: backend-модуль теперь импортируется без `ImportError` и `NameError`.
- Сохранён сценарий проверки 3x-ui и SSL по `subscription_3xui_url` и её порту, отдельно от SSH endpoint.
- Обновлены version-артефакты и transfer-документация.

## Что сломано было фактически
Предыдущий пакет падал ещё до старта API. По логам backend ломался на импорте:
- `ImportError: cannot import name 'run_xui_check' from 'src.probes'`.

После устранения этого хвоста дополнительно проявился второй backend-дефект:
- в `main.py` использовались `Dict`/`Any`, но не были импортированы.

## Что проверить после обновления
1. Backend контейнер стартует без циклического restart/crash loop.
2. Открывается web-интерфейс и загружается список серверов.
3. Кнопка «Добавить сервер» открывает форму и сохраняет запись.
4. Для сервера с `subscription_3xui_url`:
   - «Проверить 3x-ui» отрабатывает по подписочной ссылке;
   - «Проверить SSL» идёт по хосту/порту этой ссылки, а не по SSH.
5. Ручной запуск connectivity / HTTP / 3x-ui / SSL не валит backend.

## Техническая суть
Это не логическая ошибка мониторинга, а именно release-regression в backend-коде: UI мог быть уже разложен, но серверный процесс не поднимался из-за битых импортов. Поэтому внешний симптом выглядел как «интерфейс сломан / кнопки не работают», хотя корневая причина была в crash на загрузке приложения.
