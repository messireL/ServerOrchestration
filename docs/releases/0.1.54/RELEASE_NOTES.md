# ServerOrchestration v0.1.54

## Что изменено
- вкладка `Проверки` разгружена: быстрые действия вынесены в отдельный блок, а планировщик и история больше не смешаны с основной таблицей;
- таблица статусов сокращена до более понятного operational-формата: `Сервер`, `Доступ`, `Состояние`, `Детали`, `Alerts`, `Проверка`, `Ошибка`;
- добавлены локальные поиск и фильтр для экрана `Проверки`, чтобы не зависеть от фильтров inventory;
- история прогонов получила клиентские фильтры по источнику (`scheduler` / ручные) и лимит вывода (`10 / 25 / 80`);
- подписи в блоке планировщика переведены на более понятный русский вид;
- документация проекта перевыпущена и синхронизирована под `v0.1.54`.

## Зачем
Экран `Проверки` начал превращаться в комбайн: запуск проверок, scheduler, длинная таблица и история шли подряд и читались тяжело. На этом шаге задача не в косметике ради косметики, а в том, чтобы экран стал operational-инструментом и не мешал переходить к следующему функционалу.

## Что проверить
1. В UI и `/health` версия = `0.1.54`.
2. Во вкладке `Проверки` сверху есть отдельный блок быстрых действий.
3. Переключатели `Статусы / Планировщик / История / Показать всё` реально скрывают и показывают нужные блоки.
4. Поиск и фильтры внутри `Проверки` работают независимо от вкладки `Серверы`.
5. История прогонов фильтруется по `Все источники / Scheduler / Ручные` и по лимитам `10 / 25 / 80`.
6. Данные подписки 3x-ui по-прежнему отображаются на русском и без техмусора `HTTP 200` / `html embedded`, если профиль разобран.

## Дальше
- следующий шаг уже делать больше в сторону функционала: operational summary, запуск целевых действий по группе серверов, первые сценарии automation/Ansible-контура;
- при этом не раздувать снова вкладку `Проверки`: новый функционал лучше разводить по отдельным operational-экранам и сценариям.
