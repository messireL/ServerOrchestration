# Архитектура проекта

## Актуальное решение
- GitHub — основной источник кода проекта.
- Portainer — основной путь deploy/update.
- Docker Compose — обязательный формат поставки стека.
- Прямой `docker compose` на сервере — только fallback и диагностика.
- Контейнеры приложения должны быть non-root.
- PostgreSQL хранит inventory, состояние проверок, alerts, задания и историю.
- Быстрые проверки (`ping`, потом HTTP/HTTPS probe, порты, 3x-ui console/subscription) идут отдельным probe-слоем, не через Ansible.
- Ansible используется как исполнительный слой для операций изменения: apt, timezone, reboot, 3x-ui update/restart, сбор журналов и т.п.

## Актуальное уточнение v0.1.9
- UI уже разделён по вкладкам, но теперь inventory доведён до рабочего CRUD-контура.
- Раздел **Серверы** поддерживает создание, редактирование, удаление и фильтрацию серверов.
- Раздел **Группы** поддерживает создание, редактирование, удаление групп и управление связями сервер↔группа.
- Таблица связей сервер↔группа вынесена отдельно, чтобы массовые привязки не превращались в угадайку.
- Ping probe продолжает жить отдельно от inventory-форм и статусов.
- Это подготавливает проект к следующему блоку: SSH/HTTP checks и первому operational-контруру с Ansible.

## Важный нюанс Portainer
Stack из Git-репозитория в Portainer лучше использовать с уже готовыми image из registry.
Для этого в проект добавлен GitHub Actions workflow, который публикует backend image в GHCR.
